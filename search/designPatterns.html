Design Pattern

They are solutions to common coding problems in software design, it is templates on how to solve that problem and can be applied on several programming languages

Consequences: improves communication between designers and developers, it also increases system maintainability and scalability, however it may lead to code complexity

how to choose a pattern: based on the nature of the problem and the issues you are facing, in addition to the constraints of your environment, and the intent, and structure of the pattern

provide tried and tested solutions to common problems, encourage code reusability, better organization and enhance communication between developers

  • Creational Design Pattern : are concerned with how classes and objects interacts with each other
    1. Singleton
    2. Factory
    3. Builder
    4. Abstract
    5. Prototype
  • Structural Design Patterns : Concerned with classes and objects are formed and made
    1. Decorator
    2. Adaptor
  • Behavioural Design Patterns
    1. Observer
    2. Strategy
    3. Command
    4. Chain if responsibility
    5. Template
    6. Visitor

Creational

Singleton: Ensure class has only one instance, and provides a global point of access to that instance

private static singleton instance, and private constructor

Factory: it provides an interface for creating an instances of as class by overriding the factory method and produce different custom instances, which promotes loose coupling and flexibility, used commonly where the class of the object is determined at runtime

Builder: separates the construction of the complex object from its representation, allowing various representations

Prototype: used to clone object and keeping performance in mind as it doesn't make it dependent on the object classes, provides features of adding removing products at run time

Behavioural

Observer: is a one to many dependency between objects, so if that one object changed its state all its dependents will be notified and updates

It is commonly used in implementing vent handling systems

Strategy: is a family of encapsulated interchangeable algorithms, made to vary independently

useful if you have different algorithms for the same task

Template: it defines the skeleton of the algorithm and lets subclasses overrides specific steps of the algorithm without having to change the structure if the main algorithm

Command: turns request into a stand-alone object that contains all information about the request, it's useful in scenarios where queueing tasks and tracking requests history and implementing callbacks functionalities

Structural

Decorator: lets you attach a additional behaviour to the function without having to change it, rather ir is rapped with another function that adds that behaviour

Adaptor: allows two incompatible instances to collaborate by providing a bridge between them, they are often used to integrates new features with old systems without modifying the system architecture
